# 《走近科学：幽灵山庄的午夜怪影》项目设计文档

## 📋 项目概述

### 项目目标
本项目是一款以《走近科学》风格包装的教育类文字冒险解谜游戏，旨在通过悬疑剧情和互动解谜的形式，教授和考查高中《通用技术：技术与设计1》课程中"技术设计的一般过程和方法"章节的核心知识点。

### 核心理念
- **"祛魅"式教学**：用科学方法揭穿看似超自然的现象，让学生在"真相大白"的顿悟中掌握工程设计知识
- **重氛围轻美术**：通过纯代码特效（CSS动画、Canvas绘图）营造悬疑氛围，避免大量美术资源制作
- **寓教于乐**：将枯燥的技术设计理论转化为引人入胜的解谜挑战

## 🎯 教学目标与知识点映射

### 课程知识点覆盖

#### 1. 技术设计的一般过程
- **发现问题**：序章中识别村庄断桥的安全隐患
- **需求分析**：序章解谜 - 将现场线索匹配到用户需求、系统约束、问题定义
- **设计构思**：第一章 - 使用黑箱法和筛选法进行方案设计
- **详细设计**：第二章 - 绘制标准的设计草图
- **人机关系**：第三章 - 考虑生理、心理、环境因素优化设计
- **检测评价**：结局 - 生成完整的设计规格说明书

#### 2. 设计方法
- **黑箱法**（第一章）：通过输入输出关系分析系统功能，剪断噪音源保留核心功能
- **筛选法**（第一章）：将功能元（动力源、控制方式）进行排列组合，筛选最优方案
- **列举法**（隐含）：分析现有产品的优缺点进行改进
- **移植借鉴法**（隐含）：风光互补发电借鉴自然能源利用

#### 3. 设计草图绘制
- **徒手绘图要求**（第二章）：
  - 画线稳、图线清晰
  - 画长直线要移动手臂而非手腕
  - 立体图绘制：先画立方体外框 → 空间分割 → 粗线勾画轮廓
  - 圆柱体绘制：正方形画圆法

#### 4. 人机关系设计
- **生理因素**（第三章）：控制箱高度适配人体尺寸（120cm站立操作）
- **心理因素**（第三章）：避免强光直射造成眩晕恐惧
- **设计目标**：高效、方便、舒适、健康、安全

## 🎬 剧情结构与章节设计

### 序章：水鬼的索命邀请
**剧情**：偏远山村李家坳的断桥边频发落水事故，村民传言是"水鬼索命"，闪烁的"鬼火"和落水声让人不寒而栗。

**游戏阶段**：
1. **Intro（引入）**：打字机效果播放悬疑旁白，介绍案件背景
2. **Explore（探索）**：手电筒探照模式，在黑暗中点击发现三条线索
   - 🟡 断桥栏杆高度异常
   - 🔵 废弃水电站枯水期停电
   - 🟢 村民夜间看急诊需过桥
3. **Puzzle（解谜）**：档案袋祛魅翻译 - 拖拽匹配谜题
   - 将"村民夜间看急诊" → 匹配到"用户需求"
   - 将"枯水期停电" → 匹配到"系统约束"
   - 将"栏杆高度异常" → 匹配到"发现问题"
4. **Outro（结束）**：揭示真相 - 不是水鬼，是缺失的照明设施

**知识点考察**：需求分析（业务需求、用户需求、系统需求、设计约束）

### 第一章：疯老头的镇魂法器
**剧情**：村里的"疯老头"造了一个贴满符咒的"镇魂法器"挂在桥头，不仅没用反而发出凄厉怪叫。调查发现这是一个设计失败的自动路灯原型。

**游戏阶段**：
1. **Intro（引入）**：介绍疯老头和他的"法器"
2. **Blackbox（黑箱法解谜）**：
   - 输入端：风力发电机、光敏电阻
   - 黑箱：贴满符咒的控制中枢（抖动特效）
   - 输出端：破喇叭（噪音）、照明灯泡（光源）
   - **任务**：点击连线剪断喇叭，保留灯泡
3. **Screening（筛选法解谜）**：老虎机式拉霸机
   - **动力源滚轮**：市电直连 / 铅酸蓄电池 / 风光互补发电
   - **控制方式滚轮**：人工开关 / 声控 / 光敏自动控制
   - **任务**：根据需求分析（停电、无人值守）锁定正确组合
   - **正确答案**：风光互补 + 光控
4. **Outro（结束）**："镇魂法器"被改造成科学的路灯核心模块

**知识点考察**：黑箱法、筛选法、功能分析

**游戏阶段**：

### 核心组件设计

#### 1. Typewriter（打字机组件）
- **功能**：逐字显示文本，营造悬疑氛围
- **参数**：`text`, `speed`, `onComplete`
- **实现**：`setTimeout` 递归 + `useState` 控制当前索引

#### 2. Flashlight（手电筒组件）
- **功能**：鼠标/触摸位置显示径向渐变光圈，其他区域黑暗
- **实现**：`radial-gradient` + `mask-image` CSS属性
- **交互**：`mousemove` / `touchmove` 事件监听

#### 3. Glitch（故障文字组件）
- **功能**：关键词（如"鬼火"、"水鬼"）的红蓝色分离抖动效果
- **实现**：CSS `::before` / `::after` 伪元素 + `clip` + `animation`

#### 4. Scene（场景组件）
- **通用结构**：
  - `phase` 状态：`intro` → `puzzle` → `outro`
  - `AnimatePresence` 实现场景切换动画
  - Header显示档案编号和系统状态
  - 打字机效果的剧情推进

## 🎨 视觉风格指南

### 配色方案
- **主背景**：`bg-black` / `bg-slate-950`
- **文字**：`text-slate-200` / `text-slate-300`
- **强调色**：
  - 蓝色（科技感）：`text-blue-400`, `border-blue-500`
  - 绿色（成功）：`text-green-500`, `bg-green-900/20`
  - 红色（警告/恐怖）：`text-red-500`, `border-red-500`
  - 黄色（光源）：`text-yellow-400`

### 特效清单
1. **CRT滤镜**（`.crt`）：
   - 扫描线：`linear-gradient` 模拟老电视
   - RGB色彩分离：`rgba(255,0,0,0.06)` 红绿蓝叠加
   
2. **CRT闪烁**（`.crt-flicker`）：
   - 不规则透明度变化：`animation: flicker 0.15s infinite`

3. **Glitch故障**（`.glitch`）：
   - 伪元素错位 + `text-shadow` + `clip` 动画

4. **发光效果**：
   - `shadow-[0_0_20px_rgba(37,99,235,0.4)]`
   - 用于按钮、高亮元素

5. **打字机光标**：
   - 可选添加闪烁的 `|` 光标

### 动画原则
- **入场动画**：`opacity: 0 → 1` + `translateY(10px → 0)`
- **悬停反馈**：`scale(1.1)` / `border` 颜色变化
- **点击反馈**：`scale(0.9)` 短暂缩小
- **过渡时长**：`transition-all duration-300` / `duration-500`

## 🎮 交互设计

### 解谜机制

#### 1. 拖拽匹配（序章）
- **实现**：点击选中 → 点击目标 → 自动匹配
- **反馈**：
  - 选中状态：蓝色发光边框
  - 正确匹配：绿色边框 + ✓ 图标
  - 错误匹配：红色边框 + ⚠ 图标

#### 2. 连线剪断（第一章-黑箱法）
- **实现**：点击连线切换 `connected` / `cut` 状态
- **视觉**：
  - 连接：实线 + 发光效果
  - 剪断：虚线 + 灰色 + 剪刀图标悬浮提示

#### 3. 拉霸机（第一章-筛选法）
- **实现**：点击卡片循环切换选项
- **动画**：`AnimatePresence` + `y轴滑动`
- **锁定**：点击"锁定方案"后验证正确性

#### 4. 绘图修正（第二章-待实现）
- **直线判定**：Canvas绘制 + 斜率计算
- **点击步骤**：按顺序点击关键点，SVG绘制辅助线

#### 5. 参数调节（第三章-待实现）
- **滑块**：`<input type="range">` + 实时预览
- **指标面板**：显示舒适度、安全性等数值

## 📝 开发备忘

### 代码规范
- 使用 TypeScript 严格模式
- 组件使用函数式 + Hooks
- 状态管理：`useState` 为主，复杂场景考虑 `useReducer`
- 样式：TailwindCSS utility classes + 自定义CSS动画

### 性能优化
- 使用 `React.memo` 避免不必要的重渲染
- 动画使用 CSS 而非 JS（GPU加速）
- 图片资源使用 WebP 格式（如需）
- 懒加载章节组件

### 浏览器兼容性
- 目标：现代浏览器（Chrome 90+, Firefox 88+, Safari 14+）
- CSS Grid / Flexbox 布局
- CSS `mask-image` 需要 `-webkit-` 前缀

## 🎯 多结局系统设计

### 结局分级（基于表现评分）

#### S级结局 - "科学之光"（90-100分）
- **解锁条件**：
  - 所有谜题一次通过，零失误
  - 无需任何提示
  - 完成时间 < 30分钟
- **奖励内容**：
  - 金色S级调查员证书
  - 完整《村庄路灯设计需求规格说明书》
  - 专利申请证书（虚拟）
  - 解锁"完美主义者"成就
- **结局旁白**：
  > "你以无可挑剔的逻辑和严谨的科学态度，完美地诠释了技术设计的精髓。李家坳的村民们将永远铭记这位用理性之光驱散迷信阴霾的科学调查员。"

#### A级结局 - "理性调查员"（75-89分）
- **解锁条件**：
  - 少量重试（总计≤3次）
  - 使用1-2次提示
  - 完成时间 < 45分钟
- **奖励内容**：
  - 银色A级调查员证书
  - 完整设计规格说明书
  - 知识点掌握报告
- **结局旁白**：
  > "虽然过程中有些许波折，但你最终用科学方法揭开了真相。这正是技术设计的魅力——在试错中不断优化，在反思中持续进步。"

#### B级结局 - "见习调查员"（60-74分）
- **解锁条件**：
  - 多次重试（4-6次）
  - 使用3-5次提示
  - 完成时间 < 60分钟
- **奖励内容**：
  - 铜色B级调查员证书
  - 基础设计文档
  - 知识点复习建议清单
- **结局旁白**：
  > "你成功完成了调查，但过程显示你在某些知识点上还需要加强。建议重新回顾相关章节，巩固技术设计的基础理论。"

#### C级结局 - "需要进修"（<60分）
- **解锁条件**：
  - 大量错误（>6次）
  - 频繁使用提示或查看答案
  - 或跳过某些谜题
- **奖励内容**：
  - 见习证书
  - 解锁"知识点回顾模式"
  - 推荐学习路径
- **结局旁白**：
  > "调查虽然结束，但你的学习之路才刚刚开始。技术设计需要扎实的理论基础和反复的实践。建议你重新学习课程内容，再次挑战这个案件。"

### 分支选择点

#### 序章分支
- **仔细调查 vs 草率结论**：
  - 仔细调查所有线索 → 后续章节获得额外提示
  - 草率跳过某些线索 → 后续难度提升

#### 第一章分支
- **方案选择演示**：
  - 选择"市电直连" → 播放动画：枯水期停电，路灯熄灭，村民再次落水
  - 选择"声控开关" → 播放动画：野外无人，路灯无法启动
  - 选择正确方案 → 播放成功动画：路灯稳定运行

#### 第二章分支
- **绘图精度**：
  - 高精度（误差<5%） → 铁匠赞赏："这图纸画得真专业！"
  - 中等精度（误差5-10%） → 铁匠勉强接受："凑合能用吧。"
  - 低精度（误差>10%） → 铁匠抱怨："这图纸太潦草了，我得猜着做。"

#### 第三章分支
- **人机参数调整**：
  - 完美调整 → 村长舒适度100%，高度赞扬
  - 部分调整 → 村长舒适度70-90%，基本满意
  - 调整不当 → 演示安全隐患（眩晕、腰痛等）

- **弹出知识点卡片**：

### 重试与学习循环

#### 无限重试机制
- 所有谜题允许无限次重试

#### 知识点强化模式

2. **对比模式**：并排显示错误方案 vs 正确方案的后果
3. **小测验**：3道选择题巩固知识点，全部答对后可继续

#### 学习档案
记录内容：
- 每个谜题的尝试次数
- 失败时的错误类型
- 使用提示的次数
- 耗时统计
- 知识点掌握度评估

### 防Bug技巧清单

#### 1. TypeScript严格模式
```typescript
// tsconfig.json 确保开启
{
  "strict": true,
  "noImplicitAny": true,
  "strictNullChecks": true
}
```

#### 2. 组件默认值
```typescript
// ✅ 正确：有默认值
const [attempts, setAttempts] = useState(0);

// ❌ 错误：可能undefined
const [score, setScore] = useState<number>();
```

#### 3. 可选链和空值合并
```typescript
// 防御性编程
const userName = gameState?.user?.name ?? '未知调查员';
const score = gameState?.score ?? 0;
```

#### 4. 关键逻辑添加日志
```typescript
function addScore(points: number) {
  console.log(`[Score] Adding ${points}, current: ${score}`);
  setScore(prev => prev + points);
}
```

#### 5. 每个阶段都启动浏览器验证
- 不要写完所有代码再测试
- **每写完一个小功能就 `npm run dev` 看看效果**
- 发现问题立即修复，不要积累
